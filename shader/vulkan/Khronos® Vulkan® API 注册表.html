1. 简介

本文档介绍了 Khronos Vulkan API 注册表架构，并提供了有关使用注册表和脚本生成各种输出的一些其他信息，包括 C 头文件以及 Vulkan API 规范和参考页面中使用的几种类型的 asciidoc 包含文件。底层 XML 文件和脚本位于 Khronos 公共 GitHub 服务器上，网址为

https://github.com/KhronosGroup/Vulkan-Docs

目前，注册表的权威副本在默认分支中维护main。

注册表使用 Vulkan API 的 XML 表示，以及一组 Python 脚本来操作加载后的注册表。脚本依靠 Pythonetree包来解析和操作 XML。其中包括 XML 架构和验证器目标。

该架构基于之前发布的 OpenGL、OpenGL ES 和 EGL API 注册表，但与其并不完全相同。它经过扩展，可以表示这些 API 不需要的其他类型和概念，例如结构和枚举类型，以及特定于 Vulkan 的其他类型的注册信息。

从注册表生成的 Vulkan C 头文件被签入到单独的存储库中

https://github.com/KhronosGroup/Vulkan-Headers/

1.1. 模式选择

XML 模式并非完全是纯 XML。特别是，命令返回类型/名称和参数以及结构成员在包含适当信息的 C 声明的混合模式标记中描述，其中一些 XML 节点注释声明的特定部分，例如其基类型和名称。这种选择基于之前使用 SGI.spec文件格式描述 OpenGL 的经验，大大简化了人类阅读和编写 XML 以及生成面向 C 的输出。代价是，为其他语言编写输出生成器的人必须包含足够的逻辑来解析 C 声明并提取相关信息。

如果用户发现提供的 Python 脚本无法满足其需求，他们可能会编写自己的解析器、解释器和/或转换器来操作注册表 XML。我们希望我们在本文档、RNC 架构 ( registry.rnc) 和注册表 ( vk.xml) 本身的注释中提供了足够的信息来支持此类项目。如果没有，您需要澄清；如果您在使用注册表时遇到其他问题；或者如果您提出了更改和增强功能，请在 Khronos 的公共 GitHub 项目上提交问题

https://github.com/KhronosGroup/Vulkan-Docs/issues

[Registry]请在主题行中标记您的问题，以帮助我们对它们进行分类。

2. 入门

请参阅 xml/README.adoc 存储Vulkan-Docs库以获取有关所需工具链组件（例如 Python 3、g++ 和 GNU make）的信息。

安装正确的工具后，请执行以下步骤：

  • 从 Khronos GitHub查看Vulkan-Docs上面链接的存储库（链接中有说明）

  • cd到你签出的仓库的根目录

  • 切换到默认分支（main）。

  • 调用make clean ; make install ; make test

这应该会重新生成vulkan_core.h各种特定于平台的标头，将它们安装在 中，并验证标头是否正确构建。如果您在最新的存储库标记处构建，则生成的标头应与Vulkan-Headers 存储库../include/vulkan/中的最新版本相同 。

该install目标还为 中的简单扩展加载器库生成源代码../src/ext_loader/。

其他 Makefile 目标包括xml/：

  • validate-vk.xml根据 XML 模式进行验证。如果要进行重大更改，建议使用此方法。

  • Vulkan API 规范和参考页面使用的 asciidoc 包含文件是使用父目录 Makefile 中的“make generated”目标构建的，尽管它们使用此目录中的脚本和 XML。这些文件是在构建规范时动态生成的，因为它们的内容取决于构建规范以包含的确切扩展集。

如果您只想修改 API，只需更改vk.xml并运行make 即可。请参阅示例 / 常见问题解答 / 如何操作？获取一些修改 XML 的示例。

如果您想要将注册表用于生成头文件、扩展加载器和 asciidoc 包含以外的目的，或者生成除 C 之外的语言的头文件，请从 Makefile 规则以及文件 vk.xml和脚本genvk.py、reg.py和开始generator.py。脚本如下所述，它们都位于scripts 存储库根目录下的目录中。

2.1. 标头生成脚本 -genvk.py

使用脚本生成头文件时genvk.py，需要 API 名称和配置文件名称，如 Makefile 示例所示。此外，可以要求或排除特定的 API 版本和扩展。根据此信息，生成器脚本会提取相关接口并为它们创建 C 语言头文件。 包含当前Vulkan API 规范genvk.py版本的预定义生成器选项 。

生成器脚本旨在通过编写新的生成器类来推广到其他语言。此类生成器必须将 XML 中的 C 类型和定义重写为适合其语言的内容。

2.2. 注册表处理脚本 -reg.py

XML 处理在 中完成reg.py，其中包含多个对象和方法，用于加载注册表以及提取用于生成标头的接口和扩展。有一些以注释形式呈现的内部文档，但目前还没有更广泛的文档。

2.3. 输出生成器脚本 -generator.py

一旦加载了注册表，就会使用COutputGenerator中定义的类 generator.py来创建头文件。该类DocOutputGenerator用于创建 asciidoc 包含文件。可以根据需要添加用于其他目的的输出生成器。其中包括各种输出生成器：

  • cgenerator.py- 生成 C 头文件

  • docgenerator.py- 为 API 生成 asciidoc 包含内容

  • hostsyncgenerator.py- 生成主机同步表包括 API

  • validitygenerator.py- 生成有效性语言包括

  • pygenerator.py- 生成注册表部分内容的基于 Python 字典的编码，用于规范生成

  • extensionStubSource.py- 生成一个简单的 C 扩展加载器。

3. Vulkan 注册表架构

Vulkan 注册表的格式是顶级registry标签，其中包含 types、enums、commands、feature和extension描述 API 不同元素的标签，如下所述。此描述对应于正式的 Relax NG 架构文件 ， registry.rnc可以根据该文件验证 XML 注册表文件。

目前，此模式中唯一的注册表是核心 Vulkan API 注册表 vk.xml。

3.1. 个人资料

根据请求的 API 配置文件，类型和枚举可以具有不同的定义。此功能在当前的 Vulkan API 中未使用，但将来可能会使用。功能和扩展可以根据请求的 API 配置文件有条件地包含一些元素。

3.2. API 名称

特定 API 版本的功能和扩展可以使用API 名称标记为属于功能​​类别。这旨在允许同一系列中的多个密切相关的 API 规范（例如桌面和移动规范）共享相同的 XML。API 名称是任意的字母数字字符串，但应选择与相应的 API 匹配的名称。例如，Vulkan 和 OpenXR 分别使用vulkan和openxr作为其 API 名称。

api标签的属性和feature标签supported 的属性extensions必须是逗号分隔的一个或多个 API 名称列表，所有 API 名称均与该功能或扩展相匹配。从 XML 生成标头和其他工件时，可以向处理脚本指定 API 名称，从而仅选择 API 名称与指定名称相匹配的功能和扩展。

用于定义类型和类型组的其他几个标签也支持 api属性。如果存在，属性值必须是一个或多个 API 名称的逗号分隔列表。这允许为不同的、密切相关的 API 专门定义。

4. 注册表根（registry标签）

包含registry一个或多个相关 API 的完整定义。

4.1.registry标签的属性

没有任何。

registry4.2.标签内容

以下每个标签有零个或多个，通常按此顺序排列（尽管顺序并不重要）：

  • comment- 包含任意文本，例如版权声明。

  • platforms- 定义与平台特定的API 扩展相对应的平台名称。

  • tags- 定义用于扩展和层的作者 ID。作者 ID 在“Vulkan 文档和扩展：程序和约定”文档的“层和扩展”部分中有详细描述。

  • types- 定义 API 类型。通常只使用一个标签。

  • enums- 定义 API 令牌名称和值。通常使用多个标签。相关组可以标记为与 type标签对应的枚举类型，并产生 Cenum声明。此功能在 Vulkan API 中被大量使用。

  • commands- 定义 API 命令（函数）。通常只使用一个标签。

  • feature- 定义 API 功能接口（或多或少，API 版本）。每个功能集一个标签。

  • extensions- 定义API扩展接口。通常只使用一个标签，包装多个扩展。

  • formats- 定义图像格式的属性。仅使用一个标签，涵盖所有格式。

  • spirvextensions- 定义 SPIR-V 扩展与启用每个扩展的 API 接口之间的关系。仅使用一个标签，包装所有 SPIR-V 扩展。

  • spirvcapabilities- 定义 SPIR-V 功能与启用每个功能的 API 接口之间的关系。仅使用一个标签，包装所有 SPIR-V 功能。

  • sync- 定义同步对象

  • tag::syncstage - 定义所有管道阶段

  • tag::syncaccess - 定义所有访问类型

  • tag::syncpipeline - 定义管道的逻辑顺序

4.2.1. 评论标签（commentTag）

标签comment包含任意字符串，并且未使用。注释标签可能出现在架构中的多个位置，如下所述。如果注释标签原本会出现在生成的标头、asciidoc 包含文件等中，则输出生成器会删除它们。

5. 平台名称块（platforms标签）

包含platformsVulkan 窗口系统特定扩展支持的平台的平台 ID 描述。

5.1.platforms标签的属性

  • comment- 可选。任意字符串（未使用）。

platforms5.2.标签内容

零个或多个platform标签，任意顺序（尽管它们通常按平台名称排序）。

6. 平台名称（platform标签）

一个platform标签描述单个平台名称。

6.1.platform标签的属性

  • name- 必填。平台名称。这必须是与平台名称对应的短字母数字字符串，可作为 C99 标识符的一部分。最好使用小写字母。在某些情况下，可能需要将平台功能的子集与整个平台区分开来。在这些情况下，平台名称应以整个平台名称开头，后跟子_集名称。

    笔记

    例如，

    name="xlib"

    用于X Window System、Xlib客户端库平台。

    name="xlib_xrandr"

    用于xlib平台内的XRandR功能。

  • protect- 必需。这必须是一个 C99 预处理器标记，以 开头，VK_USE_PLATFORM_ 后跟平台名称（转换为大写），后跟_ 支持该平台的相应窗口系统特定扩展的扩展后缀。

    笔记

    例如，

    protect="VK_USE_PLATFORM_XLIB_XRANDR_EXT"

    用于xlib_xrandr平台名称。

  • comment- 可选。任意字符串（未使用）。

platform6.2.标签内容

不允许有内容。所有信息都包含在属性中。

7. 作者 ID 区块（tags标签）

标签tags包含authorid描述扩展和层作者使用的保留作者 ID 的标签。

7.1.tags标签的属性

  • comment- 可选。任意字符串（未使用）。

7.2.tags标签的内容

零个或多个tag标签，任意顺序（尽管它们通常按作者 ID 排序）。

8. 作者 ID（tag标签）

标签tag包含定义单个作者 ID 的信息。

8.1.tag标签的属性

  • name- 必填。作者 ID，已在 Khronos 注册。简短的大写字符串，通常是作者、项目或公司名称的缩写。

  • author- 必填。作者姓名，例如完整的公司名称或项目名称。

  • contact- 必填。使用该 ID 注册或当前负责扩展和层的联系人，包括足够的联系信息，例如个人姓名和电子邮件地址、GitHub 用户名或其他联系信息。

8.2.tag标签的内容

不允许有内容。所有信息都包含在属性中。

9. API 类型块（types标签）

标签types包含 API 中使用的派生类型的定义。

9.1.types标签的属性

  • comment- 可选。任意字符串（未使用）。

9.2.types标签的内容

零个或多个typeandcomment标签，顺序任意（尽管它们通常按将其他类型的依赖项放在列表前面的方式排序）。标签comment主要用于指示相关类型的分组。

10. API 类型（type标签）

标签type包含可用于生成与类型对应的 C 代码的信息。在许多情况下，这只是合法的 C 代码，其中的属性或嵌入标签表示类型名称和用于定义此类型的其他类型。在某些情况下，额外的属性和嵌入类型信息可用于生成更复杂的 C 类型。

10.1.type标签的属性

  • requires- 可选。此类型需要另一个类型名称来完成其定义。

  • name- 可选。此类型的名称（如果未在标签主体中定义）。

20. Formats (formats Tag)

The formats tag contains definitions of each of the image formats which are defined for the API.

20.1. Attributes of formats Tags

None.

20.2. Contents of formats Tags

A formats block contains zero or more format tags, each describing an image format, in arbitrary order.

21. Image Format (format Tag)

Image formats are described in individual format tags. An image format corresponds to a Vulkan VkFormat enumerant. This tag contains information specifying the structure and meaning of different parts of the format. The meaning of different parts of the format information is described in more detail in the “Format Definition” section of the Vulkan API Specification.

21.1. Attributes of format Tags

  • name - required. Format name, matching a VkFormat enum name. Example: name="VK_FORMAT_R8_UNORM".

  • class - required. Format class. A string whose value is shared by a group of formats which may be compatible, and is a textual description of something important that group has in common. Example: class="8-bit".

  • blockSize - required. A decimal integer which is the texel block size, in bytes, of the format.

  • texelsPerBlock - required A decimal integer which is the number of texels in a texel block of the format.

  • blockExtent - optional. Three-dimensional extent of a texel block. A comma-separated list of three decimal integers. If not present, blockExtent="1,1,1" is assumed.

  • packed - optional. A decimal integer which is the number of bits into which the format is packed. If defined, a single image element in this format can be stored in the same space as a scalar type of this bit width.

  • compressed - optional. A string whose value is shared by a group of formats which use the same general texture compression scheme, and is a textual description of that scheme. Example: compressed="ASTC LDR".

  • chroma - optional. A string used to mark if Y′CBCR samplers are required by default when using this format. Must be one of the three values "420", "422", or "444" corresponding to different Y′CBCR encodings.

21.2. Contents of format Tags

One or more component tags. The order of component tags corresponds to the memory order of components of the format. Each tag describes the size and format of that component.

Zero or more plane tags, in arbitrary order. Each tag describes the scale of a specific image plane of the format relative to the overall format.

Zero or more spirvimageformat tags, in arbitrary order. Each tag describes a SPIR-V format name corresponding to the format.

21.3. Example of a format Tag

<format name="VK_FORMAT_G16_B16R16_2PLANE_420_UNORM" class="16-bit 2-plane 420" blockSize="6" texelsPerBlock="1" chroma="420">
    <component name="G" bits="16" numericFormat="UNORM" planeIndex="0"/>
    <component name="B" bits="16" numericFormat="UNORM" planeIndex="1"/>
    <component name="R" bits="16" numericFormat="UNORM" planeIndex="1"/>
    <plane index="0" widthDivisor="1" heightDivisor="1" compatible="VK_FORMAT_R16_UNORM"/>
    <plane index="1" widthDivisor="2" heightDivisor="2" compatible="VK_FORMAT_R16G16_UNORM"/>
</format>

22. Format Components (component Tag)

The component tag contains definitions of each of the components which are part of an image format.

22.1. Attributes of component Tags

  • name - required. A string specifying the name of this component. Must be one of the values "R", "G", "B", "A", "D", or "S" corresponding to red, green, blue, alpha, depth, and stencil components, respectively.

  • bits - required. Must be either a decimal integer which is the number of bits in this component, or "compressed", corresponding to a specific compression scheme.

  • numericFormat - required. A string specifying the scalar data type of the component. Must be one of the following values:

    • "SFIXED5" - signed fractional integer values that get converted to floating-point in the range [-1024,1023.96875]

    • "SFLOAT" - signed floating-point numbers

    • "SINT" - signed integer values in the range [-2n-1,2n-1-1]

    • "SNORM" - signed normalized values in the range [-1,1]

    • "SRGB" - R, G, and B components are unsigned normalized values that represent values using sRGB nonlinear encoding, while the A component (if one exists) is a regular unsigned normalized value

    • "SSCALED" - signed integer values that get converted to floating-point in the range [-2n-1,2n-1-1]

    • "UFLOAT" - unsigned floating-point numbers (used by packed, shared exponent, and some compressed formats)

    • "UINT" - unsigned integer values in the range [0,2n-1]

    • "UNORM" - unsigned normalized values in the range [0,1]

    • "USCALED" - unsigned integer values that get converted to floating-point in the range [0,2n-1]

  • planeIndex- 可选。一个十进制整数，指定此组件位于哪个平面。如果存在，则必须与同一标签index的属性值 相对应。planecomponent

35. 示例/常见问题/我该如何做？

对于刚接触 Registry 的人来说，如何进行更改可能不是一目了然的。本节包含一些提示和示例，可帮助您通过更改 Registry XML 描述来更改 Vulkan 标头。

首先，按照所述步骤获取包含注册表的Vulkan GitHub 存储库，并组装使用 XML 注册表所需的工具。一旦您能够从 重新生成 Vulkan 标头vk.xml，您就可以开始进行更改。

35.1. 总体策略

如果要添加API，请执行以下步骤来创建该 API 元素的描述：

  • 对于每种类型、枚举组、编译时常量和添加的命令，创建适当的新的、、type或标签 来定义相关的接口。enumsenumcommand

  • 确保所有添加的类型和命令通过添加嵌套type标签适当地标记它们对其他类型的依赖关系。

  • 确保每个新标记都定义相应类型、枚举组、常量或命令的名称，并且结构/联合类型和命令标记其所有成员和参数的类型和名称。这对于自动依赖关系过程的运行至关重要。

如果您正在修改现有的 API，只需在现有标签中进行适当的更改。

添加定义后，继续下一部分以创建对更改的功能的依赖关系。

35.2. API 功能依赖关系

当您添加新的 API 元素时，它们不会导致生成的标头中发生相应的更改，除非生成的接口需要它们。这使得可以在单个注册表中包含不同的 API 版本和扩展，并根据需要将它们拉出。因此，您必须在相应的标记中引入对新功能的依赖 feature。

目前，Vulkan 定义了多个 API 版本。可以通过搜索找到初始 Vulkan 1.0 标签

<feature api="vulkan" name="VK_VERSION_1_0"

一个feature标签内有多个require标签。其中一些标签用于表达扩展交互，其他标签仅作为该 API 功能相关部分的逻辑分组机制。

35.2.1. API 功能演练

本节介绍标签中所需的前几个 API 功能 vk.xml feature，展示每个需求如何引入类型、标记和命令定义并将它们转换为 C 头文件中的定义vulkan_core.h。

考虑一下的前几行feature：

<require comment="Header boilerplate">
    <type name="vk_platform"/>

<require comment="API constants">

请注意，vk_platform并不对应于实际的 C 类型，而是自由格式预处理器包含和宏和注释的集合。大多数其他标签确实定义了特定类型并且要简单得多，但在没有其他方法时，type可以使用此方法将任意 C 注入 Vulkan 标头 。一般来说，除了像这样的特定特殊情况外，强烈不建议插入任意 C。

35.7. 如何添加位标志

位掩码由 xml 中的两种类型定义 - 掩码本身的类型和有效标志的类型。

对于此示例，假设我们要定义可以处理最多 64 个独立值的位标志，如下所示：

// Flag bits for VkExampleFlagBits

首先，需要定义用作命令或函数成员的参数的标志类型。转到标有以下内容的行：

将此定义category为"bitmask"类型。bitvalues属性标识*FlagBits定义与此类型关联的标志位的条目。

接下来，转到标有以下内容的行：

<comment>Types generated from corresponding enums tags below</comment>

这为需要它的生成器定义了一个标志位类型。category的属性"enum"标识这是一个枚举类型。

最后，转到标有以下内容的行：

    <enum value="0" name="VK_EXAMPLE_NONE"/>
    <enum bitpos="0" name="VK_EXAMPLE_FIRST_BIT"/>
    <enum bitpos="1" name="VK_EXAMPLE_SECOND_BIT"/>

35.7.1. 32 位标志

还可以使用 32 位 C 枚举类型来定义位标志。这样做与 64 位位标志大致相似，但有一些关键区别。对于此示例，假设我们想要定义与上述相同的类型，但对应于enum如下所示定义的 C 和标志类型：

typedef enum VkExampleFlagBits {
    VK_EXAMPLE_NONE
    VK_DEVICE_CREATE_VALIDATION_BIT = 0x00000001,
    VK_DEVICE_CREATE_MULTI_DEVICE_IQ_MATCH_BIT = 0x00000002;
} VkExampleFlagBits;

typedef VkFlags VkExampleFlags;

要将其添加到 xml 中，需要按照与上述相同的方式将条目添加到 XML 中，但属性略有不同：

<type requires="VkExampleFlagBits" category="bitmask">typedef <type>VkFlags</type> <name>VkExampleFlags</name>;</type>

对于枚举标志值本身的定义，位宽需要更改为"32"，或者完全省略（默认位宽为 32），如下所示：

<enums name="VkExampleFlagBits" type="bitmask">

请注意，32 位位掩码必须使用 [0,30] 范围内的整数 - C 枚举仅保证支持有符号的 32 位整数值，并且为第 31 位定义无符号值可能会改变枚举类型的大小。生成器脚本将对超出此范围的值发出警告。

注册表架构面向 C 语言 API。类型和命令使用 C 子集的语法定义，尤其是结构成员和命令参数。可以使用独立于语言的语法来表示此类信息，但由于我们正在编写 C API，因此任何此类表示都必须在某个阶段转换为 C。

标vulkan.h头是使用Python 脚本中的输出生成器对象编写的。此输出生成器专门用于 C，但脚本的设计旨在支持为其他语言编写输出生成器以及文档等目的（例如，生成与类型和命令相对应的 asciidoc 片段以用于 API 规范和参考页面）。当针对其他语言时，将类型声明转换为其他语言所需的解析量很小。但是，可能需要修改一些样板 C 文本，或按语言专门化标签，以支持此类生成器。
